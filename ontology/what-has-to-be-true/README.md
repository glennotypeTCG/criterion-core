# What Has to Be True Primitives

## Purpose

These files define the core belief conditions required for each part of the system to work.

They are not marketing pages.

They are ontology primitives used to keep:

- Messaging
- Strategy
- Product design
- Founder conversations
- LP conversations
- Capital structure
- Formation logic
- AI agents and MCP workflows

aligned around the same underlying beliefs.

## Current Primitives

| Entity | Stakeholder | File | Core Belief |
|---|---|---|---|
| FetchSignal | Idea Principal | `fetchsignal.idea-principal.md` | The Idea Principal trusts FetchSignal to safely produce credible market signal. |
| SwiftVC | LP | `swiftvc.lp.md` | LPs believe early signal can improve formation decisions and venture outcomes. |
| Axiom Foundry | Idea Principal | `axiomfoundry.idea-principal.md` | The Idea Principal believes Axiom can help create a more valuable company than they could create alone. |

## System Relationship

FetchSignal makes the idea safe to test.

SwiftVC turns strong signal into a capital-backed formation opportunity.

Axiom Foundry makes that formation process more disciplined, faster, and more likely to produce a venture-backable company than the Idea Principal could create alone.

## Important Clarification

Formation can happen without Axiom Foundry.

Axiom Foundry is not what makes formation possible in the abstract. Its claim is narrower and stronger: Axiom should make formation materially better by adding structure, operating cadence, capital discipline, and investor-readiness support.

SwiftVC plays a critical role at formation because its capital commitment can convert a validated idea from evidence into a funded company-formation moment.

## Standard Sections

Each primitive should contain:

- Ontology Type
- Entity
- Primary Stakeholder
- Canonical Belief
- Plain-English Version
- Emotional Job
- Functional Job
- Proof Required
- Failure Mode
- Messaging Guardrails
- Related Primitives
