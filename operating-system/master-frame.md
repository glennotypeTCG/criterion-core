# Criterion Operating System — Master Frame

## Purpose

This file defines the operating frame for the Criterion Group ecosystem. It is the canonical human-readable summary of how Criterion Group, Fetch Happens, Axiom Foundry, SwiftVC, and NewCos work together.

The repo is the source of truth for the system. Rendered docs, decks, diagrams, dashboards, and investor materials should reconcile back to the structured files in this repository.

## Core Thesis

Criterion Group is building a company-formation system, not a traditional accelerator, consulting firm, venture studio, or fund.

The system turns validated market signals into venture-backable companies.

## Core Questions

Every company, workstream, investment decision, and operating initiative must be evaluated through three questions:

1. Do customers want it?
2. Can we build it?
3. Can we sell it?

## System-Level Translation

| Entity | Primary System Role | Primary Question |
|---|---|---|
| Fetch Happens | Market signal engine | Do customers want it? |
| Axiom Foundry | Company formation layer | Can we build it? |
| SwiftVC | Embedded first-check capital layer | Is it worth funding, and can we own enough of the upside? |
| Criterion Group | Ecosystem orchestration layer | Can this system repeatedly manufacture venture-backable companies? |
| NewCos | Formed startups | Can they answer all three questions in the market? |

## Columns and Stages

| Column | Name | Core Question | Primary Purpose |
|---|---|---|---|
| A | Revenue Discovery | Do customers want it? | Validate demand and market signal before overbuilding. |
| B | Product System | Can we build it? | Build and operate the product in a technically and economically sustainable way. |
| C | Revenue Capture | Can we sell it? | Convert demand into revenue, funding, usage, retention, or portfolio value. |

## Constraint Rule

At any moment, each company or workstream has one primary constraint.

If a task does not relieve the current constraint, defer it.

## Canonical Happy Path

1. Idea person enters the Criterion ecosystem.
2. Fetch Happens C6 converts the idea person into a validation customer.
3. Fetch Happens C7 completes the validation study.
4. A positive signal triggers SwiftVC investment screening.
5. SwiftVC C6 reaches term sheet intent.
6. Axiom Foundry C6 scopes and co-sells the formation package.
7. The idea person signs SwiftVC investment documents and the Axiom Foundry services agreement.
8. SwiftVC C7 begins the long-tail investment lifecycle.
9. Axiom Foundry C7 begins the roughly six-month formation sprint.
10. The NewCo graduates as an independent company.
11. The NewCo raises, grows, fails, or exits while SwiftVC remains economically exposed.

## Five Controlled Systems

Criterion Core separates the operating model into five systems:

1. Brand System — how the market understands each entity.
2. Ontology System — objects, states, relationships, and vocabulary.
3. Governance System — decision rights, gates, evidence standards, and conflict rules.
4. Economic System — capital flow, ownership, services revenue, portfolio construction, and return logic.
5. Operational System — workflows, handoffs, artifacts, playbooks, and cadence.

## Master Operating Rule

Every activity must map to a customer, constraint, stage, owner, artifact, and economic reason.
